# 描画完全性・アセット管理方針（2026-07-15）

## 目的

マップ遷移、歩行、飛行、戦闘開始、会話終了のどのタイミングでも、画像の遅延昇格による透明タイル、白丸、黒背景を表示しない。個別画面ごとの再読込で直すのではなく、キャッシュ、メモリ画像、テーマデータ、描画開始条件を一つの設計として管理する。

## バックアップ比較で判明した回帰

`バックアップ.zip` の安定版は、ゲーム開始前に `GRAPHICS.data` の全キーを `GRAPHICS.images` へ読み込み、全件の完了後にゲームロジックを開始していた。現行版では開始時キーだけを読み込む方式へ変更されており、Service Workerの全キャッシュが完了していても、描画コードが同期参照する `GRAPHICS.images` に未登録の画像が残っていた。

キャッシュ済みであることと、現在のJavaScript実行環境でデコード済み画像を同期参照できることは別条件である。迷路・溶岩・ランダムテーマの戦闘背景欠けは、背景ファイルの欠損ではなく、この二層を混同したことが根因だった。

## 現行の不変条件

1. 初回確認で「はい」を選ぶ場合は全データ取得完了を待つ。「いいえ」を選ぶ場合もゲーム開始後に全データのCache Storage取得を継続する。
2. ゲーム本編開始前に、`GRAPHICS.data` 全件を `GRAPHICS.images` へ読み込み終える。マップチップ、オーバーレイ、主人公フレーム、戦闘背景を遅延昇格に依存させない。
3. 戦闘背景は `Field.getBattleBg()` が返したキーを、デコード済み画像から同期解決する。個別ファイル障害時は、デコード済みの `battle_bg_dungeon`、次に `battle_bg_field` を使用する。`backgroundImage = none` で戦闘を開始してはならない。
4. ランダムダンジョンのマップテーマと戦闘背景は、`Dungeon.randomVisualThemes` の同一レコードから設定する。古いセーブにテーマまたは背景の片方しかない場合は `ensureRandomVisualTheme()` がペアを修復する。
5. フィールド描画は静的レイヤーを4マス単位で再利用するが、主人公、円光、HUDは毎フレーム更新する。画像ロード完了後の再描画を正常系にしない。
6. ダンジョン円光は画面中央ではなく主人公のワールド座標に置き、歩行ごとに主人公座標へ追従させる。
7. 冒険者は上下左右×2歩の8キーを命名規則で管理する。方向・歩行フレームは冒険者データに保存し、会話選択中と報酬会話中は表示を維持し、報酬会話完了後に消去する。
8. ワールドの橋は基礎タイルが `W` でも、`MapRegistry.getWorldSurfaceAt()` が陸上として返す。通行、船の乗降、エンカウント種別、戦闘背景でこの判定を共有し、生のタイル文字だけで海へ戻さない。

迷路フロアは「迷路だから専用背景へ上書き」しない。選ばれた外観テーマの戦闘背景を優先する。`battle_bg_maze` はテーマ情報を持たない旧セーブなどの互換用キーとして残す。溶岩フロアだけは地形体験を一致させるため、テーマより `battle_bg_fire` を優先する。

通常階の形態と外観は `rollRandomFloorPlan()` で階層につき一度だけ抽選し、生成再試行では同じ結果を維持する。50階以降の配分は、溶岩10%、浸水10%、迷路3%、宝物庫2%、深淵25%、ランダム外観50%。ランダム外観枠は、その階で解禁済みの候補から深淵を含めて等確率で選ぶ。溶岩はイグナ火山、迷路・深淵枠は深淵テーマを使う。浸水・宝物庫・10階ごとのボス階は、地形の意味を優先して壁・床を深淵テーマへ固定する。安全用フォールバック生成でも最初の抽選結果を維持し、既定チップへ無条件に戻さない。

## 浸水フロア

41階以降は単一抽選表の浸水枠10%で浸水フロアになる。ボス階、宝物庫、迷路とは重複させず、溶岩・鍵扉・歩行NPCとも排他にする。ダンジョン壁の `W` と混同しないよう、通行可能水面は専用記号 `~` として管理する。床面は波を動かし、主人公は既存の魔法の小舟4方向画像で表示する。壁・床テーマは深淵へ固定し、エンカウント内容は深淵の階層テーブル、戦闘背景は専用の `battle_bg_flooded` を使用する。

## 光の宮殿外観

光の宮殿テーマでは汎用の灰色い瓦礫・ひび差分を床へ追加しない。壁の露出境界には `tile_light_wall_face` を描き、5列ごとに `tile_light_wall_face_prism` を使用する。固定の光の宮殿・頂の神殿と、ランダム外観枠で選ばれた光の宮殿の双方で同じ壁面規則を使う。

ランダムダンジョンの階段 `S` はテーマ側の画像定義へ依存させず、必ずテーマ床 `T` の上へ透過画像 `overlay_dungeon_stairs` を重ねる。

## ダンジョン壁面レジストリ

固定・ランダムダンジョンの露出壁面は、`map.js` の `DUNGEON_WALL_FACE_THEMES` を正本とする。各マップへ同じ `wallFaceImg` を複製せず、`themeKey` から壁面画像、アクセント画像、配置間隔、置換／オーバーレイ方式を解決する。個別マップの `wallFaceImg` は演出上の明示的な上書きが必要な場合だけ使用する。

壁面は、直下が `W` ではない境界列だけへ48px高で描画する。`W.lowerLayer` または `W.animatedWater` のテーマでは `W` が水面なので、壁面画像へ置き換えない。現状では `CRENA_CAVE` と `SEABED_TEMPLE` を `disabled: true` として明示し、誤適用を検証で失敗させる。

専用壁面は開始洞窟／深淵、イグニス火山、風の神殿、森の風穴、大灯台、雷の要塞、光の宮殿、魔王城、ガルヴァニア洞窟、闇の神殿跡、グレゼリア禁足地をカバーする。禁忌の森は例外で、単一壁面へ置換せず `TILE_THEMES.FORBIDDEN_FOREST.W.variants` の4種類を維持する。固定ガルヴァニア洞窟の全フロアは `GALVANIA_CAVE` を使用し、旧 `DEFAULT` 上書きへ戻さない。

## 生成後の完全性

非ボス階は階段タイルを必ず1個だけ持つ。生成完了後に、実配列と保存寸法の一致、主人公から全通行可能タイルへの地形接続、宝箱を障害物・鍵扉を所持鍵依存として扱った階段到達性、泉・裂け目・冒険者・試練の天使・鍵関連物の到達性と座標重複を検査する。失敗時は同じフロア抽選結果のまま再生成する。旧セーブ読込時も、寸法補正、未接続領域の除去、重複階段の整理、到達不能な特殊物の再配置を行う。

全体マップは保存メタデータだけでなく実際のタイル行列から縦横寸法を再計算し、描画領域の横倍率と縦倍率の小さい方を採用する。上下左右に余白を確保し、縦長マップの下端や横長マップの右端を切らない。

草原の追加オブジェクトは座標ハッシュによる決定論的配置を維持し、ワールド草地の出現密度を従来の約16.67%（1/6）から10%（1/10）へ下げる。

## 生成アセットの管理

冒険者の生成原版は `assets/managed/source/generated/20260715-dungeon-adventurer/` に保存する。実行時フレームは `assets/map/overlays/overlay_dungeon_adventurer_{direction}_{step}_v001.png` とする。再生成時は `tools/build-dungeon-adventurer-frames.py` を使い、4列×2行の透過済みシートから32×32の8フレームを作る。

終盤固有技10種の生成原版は `assets/managed/source/generated/20260715-ultimate-skill-fx/`、実行時画像は `assets/effect/fx_ultimate_*_v001.png` を正本とする。

## 回帰検証

- `tools/validation/validate-field-render-lifecycle.js`: 全GRAPHICS読込、背景フォールバック、溶岩背景、テーマ背景優先、円光追従、冒険者隣接・8フレーム・会話後消去、ログ保持を検証する。
- `tools/validation/validate-visual-assets.js`: 13テーマの最低階、全テーマの壁・床チップ差、テーマ背景登録、浸水水面設定、画像ファイル、固定ダンジョン画像を検証する。
- `tools/dungeon-wall-visual-qa.html`: 14系統を同一の32px床／48px露出壁条件で並べ、境界、明度、アクセント、水面除外を原寸比較する。
- `tools/validation/validate-map-safety.js`: 生成階の到達性に加え、テーマ候補数、浸水変換、船移動用水面から階段までの到達性を検証する。
- `tools/validation/validate-opening-flow.js`: 全キャッシュ確認、初期必須画像、キャッシュ世代を検証する。
- 実画面QAでは別オリジンで旧Service Workerを排除し、全GRAPHICS読込後に迷路テーマ背景と溶岩背景の要求キー・実描画キー・CSS背景URLの一致を確認する。

## 禁止事項

- 描画欠けを `Field.render()` の追加呼び出しや「一歩後の更新」で隠さない。
- 新しいマップ画像・戦闘背景をコード内の単発文字列だけで追加しない。`assets.js` と正本テーマ表へ登録し、検証へ追加する。
- 正常系で白丸、単色矩形、背景なしを表示しない。開発用マーカーは明示的なQA環境以外で使用しない。
- マップテーマと戦闘背景を別々の乱数や別々の分岐で決定しない。
