# PRISMA ABYSS Current Product Directives 2026-07-14

この文書は、過去資料に残る旧方針より優先する現行制作指示をまとめる。

## 1. セリフの文字数

- セリフに固定の文字数制限は設けない。
- 文章は、実際の会話画面での読みやすさ、人物の呼吸、情報密度、場面のテンポで評価する。
- 改行や吹き出し分割は、意味と演出を良くするために行う。数値基準を満たすための機械的な分割は禁止する。
- 旧資料の文字数超過判定と、文字数を合否条件にする検証は廃止する。

## 2. 初回全画像キャッシュ方針

- 初回起動時に、全画像データをその場でダウンロードするか確認する。
- 「はい」の場合は、件数・取得容量・進捗を表示して全量取得の完了を待ってからゲームを開始する。
- 「いいえ」の場合は、ゲームを即開始しつつ、起動後に全画像データのバックグラウンドキャッシュを必ず開始する。「いいえ」は全量キャッシュの拒否ではなく、起動前に待たない選択である。
- 画像の遅延・オンデマンド取得を基本方針にしない。マップ、モンスター、ボス、戦闘背景、戦闘エフェクト、新規追加画像を `assets.js` の一元管理された全量キャッシュ対象へ含める。
- どちらを選んだ場合も、リュミナ村、開幕会話、ジェリー戦、「始まりの洞穴」のマップチップ、通常敵、ボス、フィールド／洞窟戦闘背景は起動直後のメモリ先読みに含め、戦闘開始時の画像欠落を禁止する。
- 始まりの洞穴を攻略し、長老への報告会話 `PROLOGUE3` が終了して `storyStep: 2 / subStep: 1` へ進んだ直後に紙芝居OPを再生する。
- 紙芝居OPの終了後に全量ダウンロード案内を重複表示しない。OPはキャッシュ確認とは独立した演出とする。
- 設定画面の全データダウンロードは、失敗分の再取得や手動確認の導線として残す。
- 容量表示は実配信環境で取得できる画像サイズ合計を正とする。検証環境だけで0Bになった観察から、製品の容量計算不具合と断定しない。
- 改善対象は、対象件数の説明、進捗、失敗時の再試行、キャッシュ完全性、更新互換性とする。
- 描画負荷はキャッシュ対象を減らして解決せず、静的マップ層の再構築頻度、テクスチャ再利用、可視範囲管理を最適化して解決する。
- アセットの重複削除や安全な名前整理によって取得量を減らすことは推奨する。
- 遅延画像の読込完了、会話終了、ストーリー状態変更、マップ遷移の直後は静的マップ層を即時再構築し、一歩歩くまで地形やイベント人物が古い表示のまま残ることを禁止する。
- フィールド画面への復帰時は、不可視状態で静的層を構築してから同一処理フレーム内で表示し、空のマップや代替色が一瞬でも露出しないようにする。最初の世界地図表示に必要な基本地形も起動時にメモリへ読み込む。
- 宿屋の「転送の扉」は深淵へ初めて実際に潜入するまで非表示にする。`???` や無効ボタンも表示しない。

## 3. チュートリアル実装時期

- チュートリアルは、対象となる全画面と操作フローが完成してから作成する。
- 開発途中の古い画面を前提にしたチュートリアルは、完成時に誤情報となるため実装しない。
- 現段階では、説明すべき項目、発生候補、必要な計測だけを記録してよい。
- UI完成ゲートを通過するまで、チュートリアル本文、指差し位置、画面画像、強制操作を確定しない。
- 完成後は現行画面を使って実機検証し、そこからチュートリアルを制作する。

## 4. ガチャ

- ガチャ機能をプレイヤー向けに使用する予定はない。
- メイン、クリア後、イベントのいずれにも、ガチャ解放を予定しない。
- ガチャを仲間加入、報酬、育成、実績、チュートリアルの前提にしない。
- 既存コードやアセットは、削除リスクと未使用資産整理を比較して扱う。残っていても内部休眠コードであり、将来解放の約束ではない。
- 新しいプレイヤー導線、ボタン、解放フラグ、告知文を追加しない。

## 5. 主人公の初期ステータス評価

- 主人公マスタは `characters.js` のID `301`。
- 新規作成時は `INITIAL_DATA_TEMPLATE` を複製後、主人公マスタの職業、レアリティ、基礎ステータス等をコピーする。
- 初期武器はRank 1の武器マスタ `eid 1-6` からランダムに選ばれ、`+3`として装備される。
- 実効値は `App.calcStats()` が、基礎値、装備、配分、限界突破、スキルツリー、自己特性、装備特性、味方オーラ等を合算して算出する。
- 戦闘開始時はさらにPlayerインスタンスへ実効値を渡し、戦闘中のバフ・デバフを適用する。
- したがって、画面上の一つの数値、単一のランダム初期武器、単発の与ダメージだけから初期バランスを断定してはならない。
- 評価時は乱数を固定し、初期武器6種すべて、複数回のダメージ分布、特性適用、戦闘行動順を確認する。

## 6. アセット整理

- 完全重複は参照とキャッシュ定義を確認したうえで正本へ統合してよい。
- 日本語名、旧名、用途と異なるディレクトリは、安全に参照更新できる場合に整理してよい。
- モンスター画像はIDからの自動パス生成があるため、同一画像でも無条件削除しない。個別マッピングまたはデータ側参照を整えてから統合する。
- 変更後は構文、アセット存在、Service Worker、初回キャッシュ、開幕戦表示を検証する。
